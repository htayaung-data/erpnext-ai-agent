## How To Use
1. Run regression pack first:
   - `bash impl_factory/04_automation/bench_scripts/run_step11_regression.sh`
2. Execute UAT scenarios in `step11_uat_matrix.md`.
3. Record outcomes in `step11_uat_evidence_template.md`.
4. Attach generated logs from `impl_factory/04_automation/logs/`.
5. Use `step13_phase2_closure_report_2026-03-02.md` as the closed Phase 2 baseline.
6. Use `step13_behavioral_class_development_contract.md` and `step14_phase3_regression_discipline_contract.md` as the active rulebooks for Phase 3 and future expansion.
7. Use `step14_phase3_execution_worklist_2026-03-03.md` as the immediate work sequence for Phase 3 kickoff.
8. Use `step14_phase3_risk_tier_inventory_2026-03-03.md` to decide rerun obligations by impact.
9. Use `step14_incident_register_template.md` to record and close Tier 1 / Tier 2 incidents consistently.
10. Use `step14_phase3_incident_backfill_2026-03-03.md` as the baseline incident inventory carried forward from Phase 1 and Phase 2.
11. Use `step14_phase3_standing_browser_smoke_pack.md` as the mandatory browser parity pack after significant shared-runtime changes.
12. Use `step14_phase3_rerun_decision_checklist.md` before closing changes so reruns stay evidence-based and consistent.
13. Use `step14_phase3_weekly_quality_review_template.md` for recurring quality review.
14. Use `step14_phase3_ownership_register_2026-03-03.md` to keep asset ownership explicit.
15. Use `step15_first_approved_expansion_candidate_threshold_exception_list.md` as the first controlled expansion target after Phase 3 governance is active.
16. Use `step15_threshold_exception_list_asset_preparation_checklist.md` to gate implementation readiness for that first expansion target.
17. Use the threshold-exception-list planning assets to finish class design before any runtime implementation:

19. Use `step15_threshold_exception_list_approval_review_2026-03-03.md` as the first completed example of that approval process.
20. Use `step16_threshold_exception_list_implementation_plan_2026-03-03.md` as the approved build sequence before any runtime work starts on the first expansion candidate.
21. Use `step16_threshold_exception_list_core_slice_status_2026-03-04.md` as the current status record for the first approved threshold class slice.

26. Use `step16_phase3_operationalization_next_step_2026-03-04.md` as the current post-freeze operating instruction.
27. Use `step17_first_approved_expansion_candidate_contribution_share.md` as the formal candidate brief for the next approved class after the threshold baseline freeze.
28. Use the contribution-share planning assets before relying on runtime implementation:
   - `step17_contribution_share_asset_preparation_checklist.md`

## Phase 8 Release Governance
After producing raw canary artifacts, use Phase 8 scripts for strict release control:
1. Set runtime canary routing:
   - `assistant_engine=v3_canary`
   - `ai_assistant_v3_canary_percent=10|25|50|100`
   - rollback key: `assistant_engine=v2`
1. Shadow diff (`v2` vs `v3`):
   - `python3 impl_factory/04_automation/bench_scripts/phase8_shadow_diff.py --v2-raw <v2_raw.json> --v3-raw <v3_raw.json>`
2. Stage gate (`10%|25%|50%|100%`, first-run-only scoring):
   - `python3 impl_factory/04_automation/bench_scripts/phase8_release_gate.py --raw <canary_raw_1.json> --raw <canary_raw_2.json> --stage 10 --latency-p95-sla-ms 5000`
3. Use generated outputs for go/no-go:
   - `*_phase8_release_gate_stage*.json`
   - `*_phase8_release_gate_stage*.md`
   - `*_phase8_shadow_diff.json`
   - `*_phase8_shadow_diff.md`

## Exit Gate
1. Regression runner reports success.
2. All mandatory UAT scenarios pass.
3. Any failed scenario has a linked defect and rerun evidence.
